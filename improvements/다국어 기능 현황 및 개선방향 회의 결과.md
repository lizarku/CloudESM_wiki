## 1. 문서 목차별 회의 내용 정리

### Part 1. 현재 상태 (AS-IS) 관련 논의

#### 1.1 Properties ↔ JS 중복 문제
- **현황**: Properties와 i18n JS가 두 벌로 관리되어 중복/불일치 발생
- **문제점**: 
  - messages 파일과 sl.messages 파일 간 개수 차이 (~20개 불일치)
  - resources도 마찬가지로 위아래 건수가 다름
  - 관리 포인트가 2개라서 누락/불일치 위험 증가

#### 1.2 DB 다국어 대상 테이블 수정
- **기존 문서**: COM_CODE, COM_MENU, COM_USER_ROLE, MGR_EQP, MGR_ASSET, NATION_LOCATION
- **회의 결정**: **COM_CODE, COM_MENU만 필요**
  - 나머지 테이블(MGR_EQP, MGR_ASSET, COM_USER_ROLE, NATION_LOCATION)은 다국어 불필요
  - 좌표 정보, 자산, 장비, 권한 등은 영문화 불필요

#### 1.3 기존 키 네이밍 문제
- **현황**: 메뉴별로 키를 따서 중복이 많이 발생
- **문제점**: 시퀀스 번호 기반(예: CNF.COM.0001)으로 되어 있어 의미 파악 어려움

---

### Part 2. 개발 방향 (WHY & WHAT) 관련 논의

#### 2.1 아키텍처 결정: Properties 단일화
- **결정**: Properties 파일 **하나로 통합**
- **이유**:
  - 관리 포인트 최소화
  - 현재 JSP에서 같이 쓰고 있어서 분리할 필요 없음
  - React/Vue로 가더라도 Properties에서 추출하여 JSON 생성 가능
- **방식**: "프론트에서 JSON이 필요하면 Properties에서 추출해서 만들어 쓰자"

#### 2.2 DB 다국어 컬럼 방식 변경
- **기존 문서**: 컬럼 추가 방식 (code_name_en, code_name_ru)
- **회의 결정**: **랭귀지 컬럼 추가 방식으로 변경**
  - 기존: 컬럼마다 언어별 추가 (확장 시 컬럼 계속 추가)
  - 변경: lang_cd 컬럼 추가 후 데이터 row로 관리
  - **이유**: 언어 추가 시 컬럼이 아닌 데이터만 추가하면 됨

#### 2.3 언어 선택 방식
- **논의 내용**:
  - 브라우저 언어 따라가기 방식 가능
  - 로그인 시 언어 선택 방식
  - 사용자 계정별 언어 설정 (DB 저장)
- **결론**: 로그인 시 선택 + 헤더에서 변경 가능하도록

#### 2.4 번역 방식 결정
- **영어**: 기존 번역 내용 활용 + 추가 내용은 직접 번역
- **러시아어**: AI로 일괄 번역

### Part 3. 개발 방법 (HOW) 관련 논의

#### 3.1 키 네이밍 표준 결정
- **TYPE 정의 (3종류로 간소화)**:

| TYPE | 의미 | 용도 |
|------|------|------|
| `msg` | 메시지 (문장) | alert, confirm, 안내문 등 |
| `lbl` | 라벨 (단어) | 컬럼명, 필드명, 타이틀 등 |
| `btn` | 버튼 | 저장, 취소, 삭제 버튼 등 |

- **기존 5종(COM, Q, N, E, G) → 3종(msg, lbl, btn)으로 변경**
- **이유**: 
  - 알림(N)과 안내(G) 구분이 모호해서 혼용 가능성
  - 질문(Q)은 msg로 통합
  - 버튼은 번역 시 길이 관리를 위해 별도 분류 유지

#### 3.2 코드 형식 표준
- **소문자 사용**: 대문자 → **소문자**로 변경
- **약어 사용 금지**: 약어 대신 **단어 그대로 사용**
  - 예시: `msg.save.confirm`, `lbl.username`, `btn.save`
- **시퀀스 번호 제거**: `CNF.COM.0001` → `msg.save.confirm` 형태로 의미 있는 코드명 사용

#### 3.3 영어 하드코딩 처리 (플레이북 컴포넌트)
- **문제**: 플레이북 컴포넌트에 한글 대신 영어로 하드코딩된 것들이 많음
- **대응**: 한글 검색으로 찾을 수 없어서 별도 처리 필요
- **테스트 방법**: **러시아어 모드로 전환 후 영어/한글이 보이면 누락된 것**
  - AI로 러시아어 번역 일괄 처리 후 테스트

#### 3.4 UI 길이 처리 기준
- **다국어 대상 텍스트 길이 처리 방식**:
  - **중요 항목**: 직접 분기 처리 (UI별 맞춤 대응)
  - **일반 항목**: 동적 처리 (점점점 + 툴팁 등)

### Part 4. 작업 범위 및 일정 관련 논의

#### 4.1 작업 순서 재정의
```
1. 버전 통합 (UEBA 머지) - 1월 초
2. 메시지 코드 정리 (중복 제거, 키 규칙 적용)
3. DB 테이블/컬럼 작업 (COM_CODE, COM_MENU만)
4. 리스트 추출 (한글 + 영어 하드코딩)
5. 코드 작업 (Properties 통합)
6. 번역 작업 (병렬 진행)
7. 테스트/검증 + UI 수정
```

#### 4.2 일정
| 구분 | 기간 | 내용 |
|------|------|------|
| **파일럿 작업** | 12월 | 몇 페이지 시범 작업, 작업 방식 검증 |
| **다국어 작업** | 1월 | 실제 작업 진행 및 완료 목표 |
| **다국어 검증** | 2월 | 테스트/검증/UI 수정 |
| **타임존** | 3월 | 다국어 완료 후 진행 |

- **목표 마감**: 2월 말까지 다국어 완료 요청 있음
- **검증 기간이 작업보다 더 길 수 있음** (숨겨진 레이어 팝업 등)

#### 4.3 역할 분담
- **리스트 추출**: 1명 (충건)
- **코드 작업**: 팀별 분담 (SOAR팀, SIM팀 각 1명)
- **테스트/검증**: 별도 인원 필요 (QA팀 지원 요청 가능)
- **결정 필요**: 신입 투입 여부, QA팀 지원 요청 여부

## 4. 핵심 변경사항 요약

### 3줄 요약

1. **파일 통합**: Properties + JS 두 벌 → **Properties 하나로 통합**
2. **키 네이밍**: 5종(COM/Q/N/E/G) + 약어 + 대문자 → **3종(msg/lbl/btn) + 단어 그대로 + 소문자**
3. **DB 방식**: 6개 테이블 + 컬럼 추가 → **2개 테이블(COM_CODE, COM_MENU) + 랭귀지 컬럼 row 방식**

### 추가 결정사항
- **번역**: 영어는 기존 + 직접, 러시아어는 AI 번역
- **UI 처리**: 중요 항목은 직접 분기, 일반은 동적 처리
- **일정**: 파일럿 12월, 실제 작업 1월 완료 목표
